■ レビュー負債を解消する ― CodeRabbitが支えるAI駆動開発
Room C+D
11:00 - 11:45

レビューを通ったからと言って、バグが絶対無いわけではない
レビュー担当者の固定化がよく問題になる、時間も取られるのでここでボトルネックになりやすい
DORA 90%でAI導入済みだが、３割は信用していない、コードは沢山生成されるようになったが、その分レビューが追いつかない問題に直面、品質もやはりまだ完全に信頼できない
実際AIは有象無象のコードを作ってくれるが、最終的に人間がチェックしないといけない、アプルーブしたら本人に責任が行く
AIコード生成はかつてのコピペ問題をも彷彿とさせる時もある
レビュー負債とは、生成AIでコードの量は増えたが、その分質を担保するレビューのコストが上がった
これによりレビューの属人化やレビュー疲れも発生する
しかも、本気でレビューすると思うと数時間かかる場合も、PRの大きさにもよるが
じゃあなんでレビューに時間がかかるか、担当者ほどビジネス要件を把握しにくいため、コードを読んだだけでは良し悪しの判断がしにくい
システムの最終的な品質がこのレビューで保証されがち、でもあくまで品質の良さはテストで保証するものである
コードレビューに対する課題
    レビュー負荷が集中する
    レビュー品質の属人化
        Googleがレビューガイドライン作ってるいいのがある
        https://fujiharuka.github.io/google-eng-practices-ja/
    開発フローのボトルネック
    レビューの再定義が求められてきた
コードレビューはチームのコミュニケーションであったり学習の場で使いましょう
AIコードレビューが提供するもの
    コードのベストプラクティス
    ガイドラインに基づくレビュー
    Linterを用いたコードレビュー
逆に提供しないもの
    責任、最終的には人間が見る必要がある
AIコードレビューの種類
    CLI上で動くもの、コミット前でもレビューできるのが良い
    IDE上で動くもの、ビジュアルで利用できるのが良い
    クラウド上で動くもの、PRに対してレビューしてくれる
CodeRabbitは上記３つで利用できるよ。それぞれのものと連携して使える
CodeRabbitはAIコードレビューサービスでCLIもVSCodeも無料



■ nullの10億ドルの過ちから未来へ：Spring Boot 4とJSpecifyが描くnull安全設計
Room C+D
12:00 - 12:45

2000年台いろんなnull系のアノテーションの乱立により、どのアノテーションを使うかだったり、混在してしまうという弊害が発生
統一しようと305が始動したが、結局議論が止まってしまった
2014年にOptionalが導入され、null安全を確保しようとした、メリットも多いが限界点と注意点も出てきた、オーバーヘッドがあるので戻り値だけに設定するのが良い
2019年Kotlinが台頭してきた、nullの安全性を結構考慮した言語となっている、静的解析時からエラーを起こしてくれる、null安全オペレーターなどもある
2020年Java14でヌルポメッセージが改善された、Helpful ぬるぽ
2021年にJSpecifyのプロジェクト始動、nullアノテーションの統一を再び目指す
JSpecifyのメリット
    ビルド時の静的解析でnullawayでエラーにしてくれる
導入方法も解説してくれた、SpringBoot3では設定書くが、SpringBoot4ではデフォルトで入っている
nullawayはnullの静的解析ツールでgoogleのプラグイン、gradleに導入方法に応じて記載すると導入できる



■ 間も無くリリース Spring Boot 4！
Room G+H
13:15 - 14:00

機能Springフレームワーク7リリース
来週SpringBoot4リリースされる
大きな変更点
    モジュール化
    Jacson3がデフォルト
モジュール化について
    autoconfigを分割して再分化されるようになる
starterのパッケージ構成も変更されている、さいimportしなおせば基本はOK
特にwebはサーバmvcとクライアントrestclientが別れた、クライントのみで利用するのみとかで便利になった
移行はイニシャライザーからパッケージコピって置き換えがいいかも
不安な場合はclassicパッケージもある
jacksonも3に変更、xmlパッケージ名なくなりtool.jacksonになった
よくつかっているObjectMapperもこれに含まれる
例外もいらなくなり、特にjsonMapperが用意された
レジリエンス機能が追加
    リトライのサポートが入ったSpringRetryではなくコアのリトライを使うようになる、Spring Retyはもうメンテされなくなる
API Versioningができるようになった、ymlプロパティで設定できる
RestTestClientが追加された
BeanRegistrationが追加、if/forを使って柔軟な登録が可能になった
JSpecifyによるmill-safetyが実装された
    JSR305ベースで更に進化した、いろんなベンダーの協力を得た
    これによりnullawayで潜在的なnullぽを検出できるようになった
    springbootのnullableはでプリケーとになり、JSpecityに移行になる
    使い方は基本パッケージに@nullmarkedを付けていく
    JSpecityとnullawayの設定がおすすめgradleで設定するとできる、コンパイル時に潜在的なnullを見つけてくれる



■ AI駆動開発の最前線：GitHub Copilot Agent Mode と Coding Agent で実現する高速マイクロサービス開発 
Room G+H
14:15 - 15:00

AI生成のコードは同じプロンプトでも毎回違うコードが出てくる可能性があることは覚えておかないといけない
これはいわゆるバイブコーディングをしている、ある意味運任せのコード生成になってしまう可能性がある
やはりAIに全てを任せるのは基本的には非常に恐ろしい、ある程度大きいものはしっかりと仕様書を用意して再現性高くすることが重要
まず、機能実装の仕様書をAIに作らせて、細かくするために、実装計画プランも作らせる、基本は５段階ぐらいにわけるのが理想
より、クオリティ高く、しっかりしたものが作れる
特に大きなシステムの修正などは、必ず仕様書をAIに作らせてからやる
出された仕様を人間が見て、レビューすることが重要
実際２時間半で６０ファイルぐらい作ってくれた
githubのissueに仕様を書いてgithubcopilotをアサインしてPRを作らせるみたいな流れで作れる
上はcodingagentという、こいつは基本的にやることがかっちり決まって、特に修正などに使える
これから生成AIでのコーディングで起きることは、人がかいしたレビュー、そしてここがボトルネックになってしまう
再度、本当にそのコードは正しいかをプロンプトで問うと６０％から８０％にはなる、最後のチェックはやはり人になる、絶対に初回を信じてはいけない
マイクロソフトイグナイトのイベントでIntteliJのすごい発表があるから来週チェックしてください！
